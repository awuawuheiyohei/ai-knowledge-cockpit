### 20.1 系统开发控制概述

#### 编程语言与开发工具

| 概念 | 定义 |
| :--- | :--- |
| **机器语⾔ (Machine Language)** | 计算机遵循的由一长串二进制数字组成的指令，特定CPU芯片组有其专属机器语言。 |
| **汇编语⾔ (Assembly Language)** | 机器语言的高级替代方案，使用助记符表示CPU基本指令集，但仍需晦涩的硬件特定知识。 |
| **编译语⾔ (Compiled Language)** | 程序员使用编译器将高级语言源代码转换为专供特定操作系统使用的可执行文件。 |
| **解释语⾔ (Interpreted Language)** | 分发源代码，最终用户执行程序时自动触发解释器来执行系统上存储的源代码。 |
| **库** | 包含可重用代码的共享软件组件（如文本操作、机器学习），是开发人员提高效率的常用方式。 |
| **集成开发环境 (IDE)** | 提供单一环境供程序员编写、测试、调试和编译代码的工具。 |

#### 面向对象编程 (OOP)

面向对象编程侧重于交互中涉及的对象，系统中的对象协同工作以提供系统的功能或特性。

| 概念 | 定义 |
| :--- | :--- |
| **消息** | 对对象的通信或输入。 |
| **⽅法** | 定义对象响应消息时所执行动作的内部代码。 |
| **类** | 一组对象中公共方法的集合，定义了这些对象的行为。 |
| **实例** | 包含其方法的类的实例或示例（即对象本身）。 |
| **继承** | 一个类（父类/超类）的方法被另一个子类或对象继承。 |
| **多态性** | 对象的特性，允许对象因外部条件变化而对相同的消息或方法做出不同的行为响应。 |

#### 软件设计原则

*   **内聚性**：描述同一个类中各个方法目的之间的关联强度。
    *   **高内聚**：所有方法具有相似的目的，是促进良好软件设计的理想状态。
*   **耦合性**：对象之间的交互级别。
    *   **低耦合**：交互较少，对象更独立，更容易故障排除和更新，是更好的软件设计。

> 💡 **记忆 aid**：设计理想状态是 **“高内聚、低耦合”**（内聚像强力胶水把同类粘在一起，耦合像疏离的社交保持独立）。

#### 安全控制与输入验证

*   **保证程序**：系统生命周期中用于建立信任的正式过程，确保新应用程序中内置的安全控制机制能在整个生命周期内正确实施安全策略。
*   **输⼊验证**：在允许进一步处理之前，验证用户提供的值是否与预期匹配。
    *   ⚠️ **考试陷阱**：输入验证应**始终发生在服务器端**，不能仅依赖客户端验证。
*   **限度检查**：输入验证的一种形式，通过代码检查确保数字落在可接受范围内。
*   **转义输⼊**：将敏感字符替换为不会被系统执行但能向最终用户呈现相同内容的替代代码。
*   **合理性检查**：确保软件返回的值与指定标准匹配，且处于合理边界内的测试机制。

#### 故障状态

*   **故障安全**：故障时将系统置于高安全级别（甚至完全禁用），防止未经授权访问，直到管理员诊断并恢复。
*   **故障开放**：故障时允许用户绕过发生故障的安全控制，采取宽容态度。
    *   ⚠️ **考试陷阱**：安全系统必须设计为 **Fail-Secure**，绝对要避免 Fail-Open。

#### 系统开发生命周期模型

| 模型 | 特点 |
| :--- | :--- |
| **瀑布模型** | 视为一系列连续活动；现代迭代版本允许返回上一阶段纠正缺陷。 |
| **螺旋模型** | 封装瀑布模型多次迭代的元模型；每一环产生新原型，允许随需求变化返回计划阶段。 |
| **敏捷开发** | 强调客⼾需求，避开僵化模型，以迭代方式快速开发新功能。 |
| **Scrum** | 最流行的敏捷方法，工作组织成短期冲刺，每天举行简短Scrum会议讨论进展和障碍。 |
| **SAFe** | 在企业规模应用敏捷原则的综合框架，协调跨多团队工作并确保与战略目标一致。 |

#### 成熟度模型

| 模型 | 机构 | 核心阶段/功能 |
| :--- | :--- | :--- |
| **能力成熟度模型 (SW-CMM)** | 卡内基梅隆大学SEI | 1.初始级 2.可重复级 3.已定义级 4.已管理级 5.优化级 |
| **软件保证成熟度模型 (SAMM)** | OWASP | 治理、设计、实施、验证、运营（5个业务功能） |
| **IDEAL模型** | 软件工程研究所 | **I**nitating(启动), **D**iagnosing(诊断), **E**stablishing(建立), **A**cting(行动), **L**earning(学习) |

> 💡 **记忆 aid**：IDEAL 模型阶段首字母即为其名称本身 (**I-D-E-A-L**)。

#### 项目管理与配置管理

*   **⽢特图**：条形图，图形化显示项目和计划随时间变化的相互关系，用于规划、协调和跟踪特定任务。
*   **计划评审技术 (PERT)**：项目调度工具，判断产品规模，计算风险评估标准差，清晰显示任务间依赖关系。
*   **变更管理**：监控系统中经过批准的变更的有组织流程。三个核心组成部分：
    1.  请求控制
    2.  变更控制
    3.  发布控制
*   **软件配置管理 (SCM)**：控制软件版本并正式跟踪/控制修改。四个主要部分：
    1.  配置识别
    2.  配置控制
    3.  配置状态统计
    4.  配置审计
*   **DevOps**：将软件开发、质量保证和技术运营合并为单一运营模型，大幅减少开发、测试和部署时间。

#### 软件测试方法

| 测试类型 | 访问权限 / 视角 | 特点 |
| :--- | :--- | :--- |
| **⽩盒测试** | 有源代码访问权 | 逐行单步执行，检查内部逻辑结构寻找潜在错误。 |
| **⿊盒测试** | 无内部代码访问权 | 从最终用户角度提供输入场景并检查输出。 |
| **灰盒测试** | 有源代码访问权以辅助设计 | 测试时从用户视角分析输入输出，不检查内部工作原理。 |

*   **应⽤程序编程接⼝**：允许开发人员绕过传统网页，通过函数调用直接与底层服务交互。
*   **代码仓库**：源代码的中央存储点，提供版本控制、错误跟踪、网络托管、发布管理和通信功能。

### 20.2 创建数据库和数据仓库

#### 数据库模型

| 模型 | 特点 / 数据映射关系 |
| :--- | :--- |
| **层次型数据模型** | 逻辑树结构，一对多模型。每个节点可有0/1/多个子节点，但只能有1个父节点。 |
| **分布式数据模型** | 数据存储在多个逻辑连接的数据库中，用户视为单一实体，多对多映射。 |
| **关系型数据库** | 行和列的扁平二维表（关系），一对一映射。 |
| **面向对象关系型数据库** | 结合关系型数据库与面向对象编程的功能。 |

#### 关系型数据库核心概念

*   **基数**：表中的**行数**。
*   **度**：表中的**列数**。
*   **域**：一个属性（字段）可以采用的允许值的集合。

> 💡 **记忆 aid**：基数联想“基数=几行”，度联想“维度=几列”。

#### 数据库键与完整性

*   **候选键**：可用于唯一标识表中任何记录的属性子集（任两条记录在候选键上不会同值）。
*   **主键**：数据库设计者从候选键中选出用于唯一标识记录的键，每个表**只有一个**。
*   **备⽤键**：未被选为主键的候选键。
*   **外键**：强制执行两个表之间关系的键，确保外键必定对应另一表中仍存在的主键。
*   **参照完整性**：确保没有记录或行包含对不存在记录的主键的引用。

#### 数据库操作与事务

*   **数据库规范化**：使表符合特定范式要求，减少冗余并消除放错位置的数据。
*   **数据定义语⾔ (DDL)**：SQL组成部分，创建和修改数据库**结构**。
*   **数据操纵语⾔ (DML)**：SQL组成部分，与数据库模式内的**数据**进行交互。
*   **数据库事务**：一组离散的SQL指令，必须作为一个整体成功或失败，以确保数据完整性。

#### ACID 模型

关系型数据库事务必须具备的四个特征：

| 特性 | 定义 |
| :--- | :--- |
| **原⼦性** | "全有或全无"；任何部分失败则整个事务回滚，如同从未发生。 |
| **⼀致性** | 事务开始和完成时，数据库都必须符合所有数据库规则。 |
| **隔离性** | 事务彼此独立运行；修改相同数据时，必须一个完全完成后再执行另一个。 |
| **持久性** | 事务一旦提交必须被保留（通常通过事务日志等备份机制确保）。 |

> 💡 **记忆 aid**：ACID = **A**tomicity, **C**onsistency, **I**solation, **D**urability（首字母缩写即为其名）。

#### 数据库安全机制与威胁

*   **数据库视图**：作为SQL命令存储，将数据如同表一样呈现给用户。用于整理多表数据、聚合记录或限制用户仅访问有限子集。
*   **并发性**：预防性安全机制（编辑控制），使用“锁定”功能允许一用户更改同时拒绝其他用户访问/更改，保护完整性和可用性。
*   **脏读**：从一个未成功提交的事务中读取记录的问题，通常因未正确实现并发性导致。
*   **数据库污染**：在多级安全数据库中，将具有不同分类级别和/或需知要求的数据混合在一起的安全挑战。

#### 数据库攻击类型

*   **聚合攻击**：收集大量低级别/低价值安全项目，组合起来创建更高安全级别/价值信息的攻击方式。
*   **推理攻击**：利用人类思维演绎能力，将几条非敏感信息组合起来，获取本应在更高级别分类的信息的攻击方式。

> ⚠️ **考试陷阱**：区分聚合与推理——聚合是**数量上的积累**（量变引起质变），推理是**逻辑上的推导**（拼凑线索得出机密）。

### 数据库高级安全控制

#### 基于内容的访问控制
- **定义**：一种颗粒化对象控制形式，其访问控制决策基于被访问对象的内容或有效载荷。

#### 单元格抑制
- **定义**：隐藏单个数据库字段或单元格，或对其施加更多安全限制的概念。

#### 基于上下文的访问控制
- **定义**：通过评估整体情况来做出访问控制决策，关键因素在于每个对象、数据包或字段如何与整体活动或通信相关联。
- ⚠️ **易混淆点**：基于**内容**看的是数据本身（有效载荷），基于**上下文**看的是数据所处的环境/整体活动。

#### 数据库分区
- **定义**：将单一数据库拆分为多个部分，每个部分具有唯一且独特的安全级别或内容类型。
- **目的**：破坏聚合和推理漏洞。

#### 多实例化
- **定义**：同一个关系型数据库表中的两行或多行看起来具有相同的主键元素，但包含用于不同分类级别的不同数据的情况。
- **目的**：防御推理攻击。
- 💡 **记忆类比**：数据库的“平行宇宙”——同一个主键（身份），在不同安全级别（宇宙）中展现不同的数据（现实）。

#### 噪声与扰动
- **定义**：管理员将虚假或误导性数据插入数据库管理系统，以重定向或阻挠信息机密性攻击的技术。

### 数据库互联与NoSQL

#### 开放数据库连接
- **定义**：一个作为应用程序和后端数据库驱动程序之间代理的数据库功能。
- **作用**：允许应用程序与不同类型的数据库通信，而无需针对每种类型进行直接编程。
- 💡 **记忆类比**：数据库界的“万能翻译官”。

#### NoSQL数据库
- **定义**：一类使用关系模型以外的模型（如键值、图或文档）来存储数据的数据库。

| NoSQL类型 | 存储方式 | 特点与适用场景 |
| :--- | :--- | :--- |
| **键值存储** | 键值对（键为唯一标识索引） | 适用于高速应用程序和非常大的数据集 |
| **图数据库** | 图格式（节点=对象，边=关系） | 适用于表示任何类型的网络 |
| **文档存储** | 键+文档 | 存储信息比键值存储更复杂，采用文档形式 |

### 存储器威胁 (Section 20.3)

#### 存储器威胁概述
- **核心概念**：DBMS仅涵盖传统渠道的信息访问安全，而数据在处理过程中也会经过内存和物理介质等存储资源，必须采取预防措施保护底层资源。

#### 加密文件系统
- **定义**：一种防止攻击者绕过操作系统直接获取物理介质上数据的控制手段。
- **机制**：该系统只能通过主操作系统进行访问和读取。

#### 多级安全环境的存储控制
- **定义**：在多级安全环境中，必须为共享内存和存储资源配置适当的控制措施，以确保某一分类级别的数据无法在较低的分类级别被读取。

#### 云存储配置错误
- **定义**：在云计算环境中，存储访问控制错误可能导致严重后果，单一的默认安全设置配置错误可能会将敏感信息公开暴露在网络上。
- ⚠️ **常见陷阱**：云存储的“默认配置”通常是方便访问而非安全优先，上线前必须审查访问控制列表（ACL）。

#### 隐蔽存储信道
- **定义**：一种通过直接或间接操纵共享存储介质的方式，允许敏感数据在不同分类级别之间进行隐蔽传输的安全威胁。

### 理解知识型系统 (Section 20.4)

#### 专家系统
- **定义**：旨在汇集特定学科人类专家的累积知识，并将其以一致的方式应用于未来的决策中。

#### 专家系统核心组件

| 组件 | 定义 | 关键机制 |
| :--- | :--- | :--- |
| **知识库** | 包含专家系统已知的一系列规则 | 通过“如果/那么”语句将人类专家知识编纂成典 |
| **推理引擎** | 分析知识库中的信息以得出结论 | 利用逻辑推理和模糊逻辑技术，基于过往经验得出结论 |

#### 机器学习
- **定义**：利用分析能力，在无需人类洞察力直接介入的情况下，通过直接分析数据来学习、开发和更新活动模型。

| 学习类型 | 数据特征 | 机制 |
| :--- | :--- | :--- |
| **监督学习** | 带标签数据（包含正确答案） | 算法借此开发出可应用于未来案例的模型 |
| **无监督学习** | 无标签数据（不包含正确答案） | 算法独立开发模型（例如识别相似的登录组） |

#### 神经网络 / 深度学习
- **定义**：机器学习的一个子集，利用计算单元链来模仿人类大脑的生物推理过程，通过一系列相互反馈的计算决策最终得出预期输出。

#### Delta规则
- **定义**：用于神经网络训练和学习过程的规则。
- **机制**：算法通过从已知的正确决策反向推导，来确定计算链中每个节点的适当权重，从而使神经网络能够从经验中学习。
- 💡 **记忆关联**：Delta规则 = 反向推导 = 调整权重 = 从经验（正确答案）中学习。
