### 2.1 人员安全策略和程序

#### 人员安全基础概念
*   **人员是最弱环节**：无论物理/逻辑控制多强，人都能通过规避、绕过、破坏或禁用等方式影响控制效果。
*   **人员是最关键资产**：当接受适当培训并被激励时，人是安全工作的重要合作伙伴，而非“待解决的问题”。
*   **人员风险的全生命周期特性**：人员相关问题贯穿安全解决方案的开发、部署及持续管理全阶段（涉及用户、设计者、管理者、供应商等）。

#### 招聘与入职阶段
*   **招聘典型步骤**：创建工作描述 → 设定职位分类 → 筛选候选人 → 雇佣 → 培训。
*   **工作描述与职责**：
    *   必须涉及安全问题（如是否处理敏感材料/访问机密）。
    *   工作职责指导访问权限、许可和特权的分配。
    *   ⚠️ **需持续维护**：不仅用于招聘，需在整个生命周期中对比员工“应负责”与“实际负责”的内容。
*   **候选人筛选与背景调查**：
    *   **关键要素**：背景调查、推荐信核查、教育验证、安全调查验证。
    *   **背景调查内容**：工作/教育历史、推荐信、警察/政府记录、身份验证（指纹/驾照/出生证）、面试；部分职位含技能挑战、药物测试、信用检查、驾驶记录、性格测试。
    *   **在线调查**：可快速了解特质，但⚠️**必须严格遵守反歧视法律**，不同国家对犯罪历史调查限制不同，需先咨询法律部门。
*   **面试**：需采用标准化流程，确保公平且雇佣决策具备法律可辩护性。

#### 权限与协议管理
*   **最小特权原则**：授予用户完成工作任务所需的最小访问权限；真正应用需对资源和功能进行低级别细粒度控制。
*   **权限审计**：管理者应审计特权分配，确保员工未获得非严格必需的权限。
*   **雇佣协议**：概述规则限制、安全策略、职位细节、违规后果、试用期等；可能包含可接受使用政策（AUP）。
*   **可接受使用策略 (AUP)**：定义公司设备/资源的可接受与不可接受活动；专门用于分配安全角色及责任；违规可导致警告、处罚或解雇。
*   **保密协议 (NDA)**：保护机密信息不被现任/前任员工披露。离职时需提醒义务甚至要求重新签署。
    *   **单边 NDA**：一方共享敏感信息并保留控制权与保护。
    *   **双边 NDA (Mutual/Two-way)**：双方同意保护彼此机密信息。
    *   **多边 NDA**：三方或以上同意保护他方共享的敏感信息。
*   **非竞争协议 (NCA)**：限制一方（通常为员工）在关系终止后，于特定地理区域和期限内从事竞争活动，以保护商业秘密和客户关系。

#### 在职与离职管理
*   **入职流程**：将新员工纳入组织（签署协议、介绍、培训、社会化定向）；通过IAM系统创建账户并分配权限；角色/特权变化时也涉及此流程。
*   **职责漂移与权限蠕变**：员工持有的超额权限会增加组织风险，需定期审查并调整权限以重新对齐最小特权原则。
*   **强制休假**：主要用于同行审查；要求员工每年1-2周离开办公室且无远程访问，由审计员代行职责，以检测滥用、欺诈或疏忽。
*   **共谋 及防范**：多人共同实施犯罪。
    *   *防范措施*：分离职责、限制职责、强制休假、岗位轮换、交叉培训；严格监控管理员/root等特权账户。
*   **敏感/关键职位重新评估**：需定期进行与入职时同样彻底的背景调查，或执行具体检查确认资格一致性及新出现的取消资格信息。
*   **用户和实体行为分析 (UBA / UEBA)**：
    *   **UBA**：分析用户、主体、访客、客户的行为。
    *   **UEBA**：将分析扩展到实体活动（设备、系统、网络、应用），可关联漏洞、侦察、入侵、数据泄露等。监控信息用于改进人员安全策略与程序。
*   **离职流程**：从IAM系统移除身份（也适用内部转岗）。
    *   **账户处理原则**：先停用账户保留数月用于审计，无问题后再删除。⚠️过早删除会导致安全事件追踪困难。
    *   **解雇流程关键要求**：安全部与HR紧密合作；终止会议至少一名证人；告知责任限制（基于协议/NDA）；收集公司财产；立即移除/停用账户；安保陪同收集个人物品；通知安保禁止前员工无陪同进入；流程需私人尊重但做好预防。
    *   ⚠️ **禁忌时机操作**：正式通知前**不得**要回设备、停用账户、封锁PIN/智能卡、撤销停车证、分发新组织架构图、安排新员工到岗、泄露裁员信息，否则会提前警告员工。

#### 第三方与供应商管理
*   **多方风险**：多实体参与项目时，因目标、期望、时间线、预算和安全优先级差异产生的风险。需建立风险管理治理机构监督，执行一致的安全参数。
*   **服务级别协议 (SLA)**：确保服务提供商维持约定服务水平，适用于关键组件，包含财务和合同补救措施，是风险减少和避免的重要部分。
*   **供应商管理系统 (VMS)**：帮助管理和采购人员服务、软硬件等。安全方面保持通信/合同机密性、要求加密和认证交易、维护活动日志。能提升绩效、降低成本、管理风险、维持合规。

### 2.2 理解并应用风险管理概念

#### 风险管理基础
*   **风险管理的定义**：识别损坏/泄露资产的因素，结合资产价值和对策成本评估，实施成本效益高的解决方案以减轻或降低风险的详细过程。
*   **核心目标**：将风险降低到可接受水平。⚠️**通常无法实现完全无风险环境**。
*   **两大主要要素**：
    1.  **风险评估/分析**：检查环境风险，评估威胁发生可能性及损害严重程度，评估对策成本。
    2.  **风险响应**：通过成本效益分析评估对策，向高级管理层提供选项建议，根据决策实施并整合至安全策略。
*   **风险意识**：提高组织内风险知识的努力（了解资产价值、盘点威胁、明确响应），帮助认识遵守安全策略的重要性及失败的后果。

#### 核心术语与定义
| 概念 | 定义 |
| :--- | :--- |
| **资产** | 用于业务流程或任务的任何事物；组织依赖的有形或无形的人、地点、事物。 |
| **资产评估** | 基于资产对组织的重要性、关键流程使用情况、实际成本及非货币成本赋予的价值。 |
| **威胁** | 可能导致组织或特定资产产生不良/不想要结果的潜在事件（含有意/无意、内部/外部）。 |
| **威胁代理** | 故意利用漏洞的实体（通常是人，也可为程序、硬件或系统），又称攻击者、对手。 |
| **威胁事件** | 意外发生或有意利用漏洞的事件（自然或人为）。 |
| **威胁向量** | 攻击或威胁主体接触目标造成伤害的路径或方式（又称攻击向量）。 |
| **漏洞** | 资产中的弱点或防护措施的缺失/不足（缺陷、空子、疏忽、错误、限制、脆弱性）。 |
| **暴露** | 因威胁而导致资产丢失的可能性，表示漏洞有可能/将会被威胁代理或事件利用。 |
| **风险** | 威胁利用漏洞造成伤害的可能性/概率及可能导致的损害程度。 |
| **防护措施** | 任何消除或减少漏洞或防御特定威胁的事物（又称安全控制、保护机制、对策、风险响应）。 |
| **攻击** | 威胁主体故意利用漏洞造成资产损害、丢失或泄露的行为。 |
| **入侵** | 威胁主体绕过或破解安全机制；⚠️**入侵是成功的攻击**。 |
| **危害** | 有能力对组织、资产、个人或环境造成伤害、损失、损坏或不利后果的潜在源或情境。 |

> 💡 **记忆 aid - 风险要素关系链**：
> 威胁利用漏洞 → 导致暴露 → 暴露即为风险 → 风险通过防护措施缓解 → 防护措施保护被威胁危及的资产。
> 
> 💡 **记忆 aid - 风险公式**：
> 风险 = 威胁 × 漏洞
> 风险 = 危害的可能性 × 危害的严重性

#### 风险评估方法与资产估值
*   **风险评估两种方法**：
    *   **威胁优先**：考虑更广泛有害问题，但⚠️可能涉及无关威胁。
    *   **资产优先**：覆盖所有组织资源，但⚠️可能评估低价值低风险资产。
*   **资产估值的作用**：确定对业务运营的重要性/关键性，资产价值直接指导防护措施的级别。
*   **资产估值的目标**：为资产分配包含有形和无形成本的具体货币价值。⚠️确定确切价值通常困难甚至不可能，但**必须确定一个具体价值**以进行定量数学计算。
*   **资产估值关键考虑因素**：
    *   **有形成本**：购买、开发、维护、获取、防护成本等。
    *   **无形因素**：对所有者/用户/竞争对手的价值、知识产权、市场估值、替换成本、生产力影响、运营成本、责任、有用性、与研发的关系等。

### 风险评估团队与责任

#### 跨部门团队组成原则
风险评估与分析必须由**跨部门团队**执行，而非单个个体。团队成员无需均为安全专业人员或网络/系统管理员，成员多样性有助于广泛识别和应对威胁与风险。

#### 外部风险顾问的作用及工具
因风险评估复杂，许多组织会聘请风险管理顾问。顾问通常使用风险评估软件，可简化任务、提升结果可靠性、生成符合保险机构及董事会等要求的标准化报告。

#### 风险管理中的责任划分
- **高层管理人员**：承担风险管理的**主要职责**，负责决定组织实施哪些响应措施。
- **IT和安全部门人员**：通常被分配执行风险分析和风险响应建模的任务。
- **风险评估团队**：向高层提交结果作为建议。

### 行为标准与风险普遍性

#### “谨慎行为”与“合理行为”
| 行为类型 | 定义与特点 | 法律/标准定位 |
| --- | --- | --- |
| **谨慎行为** (Prudent actions) | 高度谨慎、小心和远见；主动且勤勉；常采取额外预防措施 | 高于平均水平的谨慎和勤勉；在需要更高注意标准时保护免受责任 |
| **合理行为** (Reasonable actions) | 符合普通谨慎和判断力的人的做法；基于合理、理性 | 符合普遍预期和社会规范；评估行为是否符合**最低可接受阈值**的基准 |

⚠️ **易混淆点**：谨慎行为是“高于平均水平”的主动防御；合理行为是“符合社会预期”的最低及格线。

#### 风险的普遍性与可接受性
- 所有IT系统、组织、员工执行的任务都存在风险，**无法100%消除**。
- 高层管理需决定哪些风险可接受，取决于：详细的资产和风险评估、组织预算、内部专业知识、业务条件等内外部因素。

### 风险评估基础概念

#### 风险评估的“范围”
- **定义**：指风险管理过程、项目或评估的程度或边界，定义包含和排除的内容。
- **重要性**：明确范围是有效风险管理的关键，有助于优先排序、高效分配资源、减少歧义并确保相关方共识。

#### 威胁列表
风险识别需创建组织资产所有可能威胁的详尽清单，包含威胁代理与威胁事件，且需考虑广泛的威胁来源。

### 风险评估方法

风险评估有两种主要方法：**定量风险分析**（分配实际美元数字，基于数学计算）和**定性风险分析**（分配主观和无形价值，考虑观点、直觉等）。

#### 定性风险分析
- **特点**：基于场景、感知或直觉；通过相对规模排名威胁以评估风险、成本和影响。
- **常用方法**：头脑风暴、故事板、焦点小组、调查、问卷、检查表、一对一会议、访谈、场景、Delphi技术。
- **场景技术**：单一主要威胁的书面描述（威胁如何发起及影响）；针对每个场景描述若干保障措施；参与者分配威胁级别、损失潜力和保障优势（如高/中/低、1-10）；参与者数量和多样性提升有效性。
- **Delphi技术**：匿名反馈和响应过程，用于达成群体匿名共识；目的是引出诚实、无影响的响应，最小化偏见和歧视；核心是基于**想法本身的价值**评估，而非来源。

#### 定量风险分析
- **特点**：产生具体概率指示或相对风险潜力的数字指示，结果包含货币数字。
- **核心步骤**：
  1. 编制资产清单，分配资产价值（AV）
  2. 列出每项资产可能面临的所有威胁，形成**资产-威胁组合**
  3. 计算每个资产-威胁组合的暴露因子（EF）
  4. 计算每个资产-威胁组合的单一损失期望（SLE）
  5. 计算每个威胁的年度发生率（ARO）
  6. 计算年度损失期望（ALE）得出总损失潜力
  7. 研究控制措施，计算ARO、EF和ALE的变化
  8. 进行成本/收益分析，选择最合适的应对措施

🧠 **记忆 aid - 定量分析核心公式链**：AV → EF → SLE → ARO → ALE
- SLE = AV × EF
- ALE = SLE × ARO

### 风险响应与风险类型

#### 风险响应类型
| 响应类型 | 定义 | 关键点 |
| --- | --- | --- |
| **缓解/降低** | 实施安全控制减少漏洞或阻断威胁 | 最常见的主动响应 |
| **分配/转移** | 将损失责任转嫁给其他实体 | 如购买保险 |
| **避免** | 选择低风险替代方案 | 彻底消除风险源 |
| **接受** | 控制成本超过损失，需**书面声明** | ⚠️ 必须有书面声明 |
| **威慑** | 用威慑手段阻止威胁主体攻击 | 如审计、摄像头、警告横幅 |
| **拒绝/忽略** | 不可接受的响应，可能构成疏忽 | ⚠️ 考试中绝对错误选项 |

#### 风险状态分类
- **固有风险**：实施风险管理前的自然、原声或默认风险（别名：初始风险/起始风险）。来源：供应链、开发运营、系统设计架构、组织知识技能。
- **总风险**：无安全控制时组织面临的风险。概念公式：总风险 = 威胁 × 漏洞 × 资产价值
- **控制差距**：安全控制降低的风险量。概念公式：控制差距 = 总风险 - 残余风险
- **残余风险**：实施安全控制后剩余的风险，是管理层选择接受而非缓解的风险。概念公式：残余风险 = 总风险 - 控制差距
- **控制风险**：引入安全控制带来的新风险（因控制本身存在漏洞），故风险管理需迭代。

⚠️ **易错点**：残余风险是管理层“接受”的，而控制风险是控制本身“带来”的新风险，两者概念不同。

### 安全措施评估与选择

#### 成本效益计算核心公式
- **安全措施价值** = （预安全措施 ALE - 后安全措施 ALE）- 年度安全措施成本（ACS）
- 结果为正：财务上可行（值越大越划算）；结果为负：不可行。

#### 成本有效性原则
- 保护资产的支出**不应超过**资产价值或预安全措施ALE。
- 若无成本有效的安全措施，接受风险可能是唯一合理选择。

#### 对策选择的综合考量因素
除成本效益外，还需考虑：
1. 法律责任、应尽职责/应尽关注
2. 实际情况：预算限制、系统兼容性、员工技能、产品可用性、政治因素、合同要求、市场趋势等
3. 对策核心依据：成本低于资产价值及对策收益；使攻击成本大于攻击者预期收益；解决真实已识别的问题；收益不依赖保密性；收益可测试验证；提供一致统一的保护；减少依赖以降低级联故障风险；初始部署后需最少人工干预；具备防篡改特性；仅允许特权操作者进行覆盖操作；提供失效安全/失效保护选项。

### 安全控制分类

#### 按性质分类（三种核心分类）
1. **行政控制**：含组织安全政策及法规要求的策略、程序等，聚焦人员监督与业务实践。
2. **技术/逻辑控制**：用于管理访问及保护IT资源的软硬件机制。
3. **物理控制**：保护设施及实物资产的机制。

#### 按功能分类
| 控制类型 | 功能定义 | 考点/区分 |
| --- | --- | --- |
| **预防控制** (Preventive) | 阻止或停止不希望或未授权的活动发生 | 事前阻止 |
| **检测控制** (Detection) | 发现或检测不希望或未授权的活动 | 事中发现 |
| **纠正控制** (Corrective) | 事件发生后将系统恢复至正常状态、纠正问题 | 事后修复 |
| **恢复控制** (Recovery) | 纠正控制的延伸，处理更严重破坏事件、恢复资源/功能 | 严重破坏后的恢复 |
| **威慑控制** (Deterrent) | 阻止安全政策违规行为 | ⚠️ 依赖于**说服**个体不采取行动（与预防的区别） |
| **指导控制** (Directive) | 引导、限制或控制主体行为以强制/鼓励遵守安全策略 | 规范行为 |
| **补偿控制** (Compensating) | 补充或替代现有控制以强化安全政策执行 | 替代或增强主控制 |

### 安全控制评估 (SCA) 与监控

#### 安全控制评估 (SCA)
- **定义**：对照基线或可靠性预期，对安全基础设施的各个机制进行的正式评估。
- **目标**：确保机制有效性；评估风险管理流程质量与全面性；生成优劣势报告；确认维持既有有效性或识别缺陷。
- **参考标准**：联邦机构通常基于 **NIST SP 800-53 Rev.5** 实施。所有组织均应采用其理念。

#### 监控与测量
- 安全控制需能持续监控和测量其收益。若收益无法量化、评估或比较，则实际未提供安全保障。
- 监控与测量是成本/效益分析的组成部分，需通过显著的安全改进来证明新控制部署费用的合理性。

### 风险报告与持续改进

#### 风险报告
- **核心要求**：准确、及时、全面覆盖组织、清晰精确支持决策、定期更新。
- **内部风险报告**：面向内部利益相关者，支持内部决策、风险缓解及管理。内容包含：详细风险评估、风险暴露、控制措施、管理策略有效性（如风险登记册、风险热图、KRIs、评估结果）。
- **外部风险报告**：面向外部利益相关者，提供透明度与披露。内容聚焦高层信息：风险暴露、管理政策、对财务/运营的潜在影响。

#### 风险管理工具
- **风险登记册/日志**：记录所有已识别风险的文档。用于跟踪全流程，兼具项目管理工具、历史记录、可共享整合经验的作用。
- **风险矩阵/风险热图**：在基本图表上进行的评估形式（常被标记为定性风险评估），最简单为3×3网格，比较概率和损害潜力。

#### 持续改进与成熟度
- **时间点性质**：风险分析/评估是“时间点”指标，因威胁和脆弱性不断变化，需定期重新进行。
- **安全方案更新**：安全总是变化，若所选对策无持续改进路径，应替换为提供可扩展安全改进的对策。
- **风险成熟度模型 (RMM)**：评估企业风险管理程序的关键指标和活动。
  - **1. 临时**：混乱起点
  - **2. 初步**：松散尝试，各部门可能独特执行
  - **3. 定义**：全组织采用共同或标准化风险框架
  - **4. 集成**：风险管理集成到业务流程，用指标收集数据，风险纳入业务策略决策
  - **5. 优化**：关注实现目标而非仅反应外部威胁，战略规划旨在业务成功，教训重新整合

🧠 **记忆 aid - RMM五级别**：临（临时）初（初步）定（定义）集（集成）优（优化）

### 遗留风险与生命周期

#### 遗留风险的核心关注对象
常被忽视的领域是**遗留设备**，其可能处于 EOL 或 EOS/EOSL 状态。

#### EOL 与 EOS/EOSL
- **EOL (生命周期结束)**：制造商不再生产某产品的时间点。此后可能仍提供一段时间服务与支持，但不再销售或分发新版本。EOL产品应在故障或到达EOS/EOSL前安排更换。
- **EOS/EOSL (服务支持结束/服务生命周期结束)**：有时EOL被视为等同于EOSL。

⚠️ **易错点**：EOL只是停止生产/销售，可能还有短暂支持；EOSL才是彻底停止支持。到达EOL就应计划更换，不要等到EOSL。

### EOS/EOSL 与风险框架

#### EOS/EOSL（服务支持结束/服务生命周期结束）
*   **定义**: 指供应商不再为系统提供更新和支持的状态。
*   **风险**: 继续使用此类系统会因无法修复未来漏洞而面临高妥协风险。
*   ⚠️ **Exam Trap**: 考试中常将EOSL系统作为“可接受风险”的干扰项，记住：无法修补的漏洞等同于不可接受的高风险。

#### 风险框架与 NIST 标准
*   **风险框架定义**: 用于指导风险评估、解决与监控方式的指南或方法。

| 框架名称 | 建立时间 | 适用对象 | 核心阶段/功能 | 关键特征 |
| :--- | :--- | :--- | :--- | :--- |
| **NIST RMF** (风险管理框架) | 2010年 | 美国联邦机构（强制要求） | **7个阶段**: 准备、分类、选择、实施、评估、授权、持续监控 | 用于管理信息系统和网络的信息安全与网络安全风险，建立安全基础设施及持续改进流程 |
| **NIST CSF** (网络安全框架) | 2014年 | 关键基础设施和商业组织 | **6个功能**: 识别、保护、检测、响应、恢复、治理 | ⚠️ **不是检查清单或程序**，是需持续执行的运营活动，属于改进系统 |

*   🧠 **Memory Aid (NIST CSF 6大功能)**: **I**dentify (识别) → **P**rotect (保护) → **D**etect (检测) → **R**espond (响应) → **R**ecover (恢复) → **G**overn (治理)。口诀：**I P D R R G** (I Protect Data, Recover Right, Govern!)。注意“治理”是统筹全局的功能。

### 社会工程学

#### 核心概念
*   **定义**: 利用人性与人类行为的攻击形式。人员因易犯错、被欺骗或故意违规成为安全薄弱环节。
*   **攻击利用**: 人的基本信任、助人欲望或炫耀倾向。
*   **主要形式**: 说服他人执行未授权操作，或说服他人泄露机密信息。
*   **主要目标**: 获取信息泄漏或安全环境的逻辑/物理访问权限。
*   **核心防御策略**: 用户教育和意识培训（包含角色扮演、攻击案例学习、入职及定期培训、培养警觉性与合理怀疑态度）。

#### 社会工程学心理学原则
*   **权威原则**: 利用人们对权威的服从倾向，攻击者伪装成有内部/外部有效权威的形象。
*   **恐吓原则**: 权威原则的衍生，通过权威施压、自信姿态或伤害威胁迫使目标服从，利用目标对无明确操作指南情况的不确定性。
*   **共识原则**: 利用人们模仿他人行为的自然倾向，让受害者相信某行动符合社会规范或既往案例。
*   **稀缺原则**: 强调物品或机会的稀缺性以提升其感知价值（常与紧急性原则结合）。
*   **熟悉原则**: 利用人们对熟悉事物的天然信任，伪装共同联系或模仿已知实体。
*   **信任原则**: 通过与受害者建立关系获取信任，再利用该信任让受害者违规。
*   **紧急性原则**: 制造“必须立即行动”的紧迫感让受害者来不及理性思考（常与稀缺原则配合）。
*   🧠 **Memory Aid**: 记忆七大原则首字母缩写 **A I S C U T E** (听起来像 "A cute" 诈骗)：**A**uthority (权威), **I**ntimidation (恐吓), **S**ocial consensus (共识), **S**carcity (稀缺), **F**amiliarity (熟悉/用U代替), **T**rust (信任), **U**rgency (紧急)。

#### 信息收集与欺骗技术
*   **诱导信息**: 从系统或人员中收集信息的活动，作为研究方法以构建更有效的托词。
    *   *防御*: 信息分类、控制敏感数据流动、警惕滥用企图、人员培训及向安全团队报告可疑活动。
*   **前置词**: 在通信开头/头部添加术语、表达或短语，用于细化或建立社会工程攻击的托词。
    *   *技术欺骗应用*: 插入伪造的邮件头值（如 `X-Spam-Category: LEGIT`）欺骗垃圾邮件过滤器、反恶意软件、防火墙及IDS等安全机制。

#### 网络钓鱼及其变体

| 攻击类型 | 定义/特征 | 关键细节/目标 |
| :--- | :--- | :--- |
| **网络钓鱼** | 专注于从潜在目标窃取凭证或身份信息 | 滥发邮件、伪造发件人、伪造网站链接、恶意附件 |
| **驱动下载** | 用户访问网站时无需知情即可自行安装的恶意软件 | 利用浏览器或插件漏洞 |
| **Smishing** | SMS phishing (短信钓鱼) / SPIM | 回复触发cramming(虚假收费)、链接指向恶意站点、编造借口引发对话、陌生号码高额费用；也含MMS/RCS/WhatsApp等 |
| **Vishing** | Voice-based phishing (语音钓鱼) / SpIT | 常用VoIP技术，**可伪造来电显示**（如冒充本地前缀）⚠️ **始终假设来电显示不可信** |
| **鱼叉式钓鱼** | 针对特定群体定制的钓鱼攻击 | 更具针对性 |
| **商务电子邮件破坏 (BEC)** | 鱼叉式钓鱼变体，也称CEO欺诈 | 伪装成CEO或高层，**主要目标: 说服会计/财务人员转移资金或支付发票** |
| **网络捕鲸** | 鱼叉式钓鱼的一种形式 | 目标为特定高价值个体（CEO/C级高管/管理员/高净值客户） |

*   **网络钓鱼 - 用户防御**: 怀疑意外邮件、不打开意外附件、不邮件分享敏感信息、不点击消息链接（通过书签/搜索访问网站）、报告并删除虚假信息。
*   **网络钓鱼 - 企业防御**: 考虑/屏蔽员工访问个人邮件和社交网络的风险；使用**钓鱼模拟**评估员工抵御能力（点击者需额外培训）。
*   **鱼叉式钓鱼防御**: 1. 对信息/数据/资产按价值/敏感性标注标签；2. 培训人员根据标签正确处理；3. 对异常/偏离流程/高风险操作要求澄清/确认。

#### 其他社会工程攻击

| 攻击类型 | 定义/特征 | 防御/关键点 |
| :--- | :--- | :--- |
| **垃圾邮件** | 不受欢迎/未经请求的电子邮件（含恶意内容/攻击载体） | 邮件垃圾过滤器（检查头、主题、内容关键词，丢弃/隔离/阻止） |
| **肩窥** | 物理世界/面对面，观察用户键盘或显示器 | 划分敏感区域锁门、显示器避免朝外、不公共空间处理敏感数据、隐藏密码字符、屏幕滤镜 |
| **发票诈骗** | 呈现虚假发票并强力诱导付款 | 窃取资金 |
| **恶作剧** | 说服目标执行导致问题或降低IT安全的操作 | 声称有威胁，诱导删除文件/改配置/装假杀毒软件 |
| **冒充** | 采取他人身份（伪装/欺骗/身份欺诈） | 可通过当面/电话/邮件/登录他人账户实施 |
| **尾随** | 未授权实体在合法员工**不知晓**下进入设施 | ⚠️ 员工未注意身后，门未关紧时潜入 |
| **搭便车** | 未授权实体**欺骗**合法员工获得其**同意**进入 | ⚠️ 与尾随的区别在于：受害者是否**知情同意** |
| **诱饵攻击** | 放置USB/光盘/钱包等“诱饵”诱导员工插入电脑 | 物理入侵不可行或失败时的替代技术，自动感染系统 |
| **垃圾箱搜寻** | 挖掘垃圾/废弃设备/废弃地点获取信息 | 销毁政策的重要性 |

#### 网络与UI欺骗攻击
*   **错拼域名攻击**: 预测URL拼写错误并注册对应域名，将流量导向自身（目的：竞争或恶意）。
*   **URL劫持**: 展示看似知名的链接/广告，点击后重定向至替代位置。手段：利用SEO提高排名，或借广告软件替换合法链接。
*   **点击劫持**: 将用户点击/选择重定向至恶意目标。手段：修改网页代码插入替换脚本，或添加不可见覆盖层/框架/图像映射。

#### 宏观社会工程与混合战争
*   **影响活动**: 试图引导、调整或改变公众舆论的社会工程攻击，多由国家发动，关联虚假/误导性内容传播。
*   **混合战争**: 国家将传统军事策略与现代能力（社会工程、数字影响活动、心理战、政治战术、网络战）结合，也称非线性战争或非正规战争。

### 建立和维护安全意识、教育和培训计划

#### 意识、培训与教育的层级区分

| 层级 | 目标/定义 | 受众与特征 |
| :--- | :--- | :--- |
| **安全意识** | 将安全置于首要位置，建立全组织通用的**安全理解基线** | 聚焦所有员工必须理解的关键/基础安全议题；需新鲜、有创意且常更新；结合企业文化理解 |
| **安全培训** | 教授员工执行工作任务及遵守安全策略 | 针对相似工作职能的员工群体；组织主办 |
| **安全教育** | 学习远超工作任务实际所需知识的详细努力 | 常与认证/晋升相关；安全专业职位典型要求；需掌握广泛知识（非仅具体任务） |

#### 培训计划优化与评估
*   **改进技术**: 改变培训目标焦点、调整主题顺序/重点、使用多样化呈现方法、开展角色扮演。
*   **安全冠军**: 在项目中通过同行领导、行为示范及社交鼓励，推动安全知识与实践的人员。⚠️ 通常是**非安全岗位员工**，负责将安全概念整合到团队工作中，适用于各部门。
*   **游戏化策略**: 将游戏元素融入安全合规与行为改变；提升参与度、知识应用效果、理解深度、流程效率。
*   **定期内容审查**: 确保与业务目标、组织使命及安全目标一致；借此调整焦点、增减主题、整合新技术。
*   **有效性评估方法**:
    *   ⚠️ **不能仅以参与或完成记录为准**。
    *   培训后立即测试 + 3-6个月后跟进测试（验证知识留存）。
    *   审查事件日志（对比培训前后员工相关安全违规发生率/趋势变化）。
    *   **有效标志**: 用户行为改善、后续测试高分、员工相关安全事件管理成本可衡量降低。
