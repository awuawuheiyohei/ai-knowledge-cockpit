# 第15章 安全评估与测试

### 15.1 构建安全评估和测试方案

#### 核心概念辨析
| 概念 | 定义 | 核心目的 |
| :--- | :--- | :--- |
| **安全测试** (Security Testing) | 通过自动化扫描、工具辅助渗透测试及人工尝试破坏等方式，验证安全控制措施是否正常发挥作用。 | 验证控制措施有效性 |
| **安全评估** (Security Assessments) | 对系统或应用的安全性进行全面审查，由专业人员执行风险评估以识别漏洞并提出修复建议。 | 全面审查与风险识别 |
| **安全审计** (Security Audits) | 由独立审计师执行的评估，主要目的是向第三方证明组织安全控制措施的有效性。 | 向第三方证明有效性 |

⚠️ **易混淆点**：安全测试侧重于"验证"（动手试），安全评估侧重于"审查与建议"（全面体检），安全审计侧重于"独立证明"（出具证明给第三方看）。

#### 审计类型
| 类型 | 执行者 | 特点 |
| :--- | :--- | :--- |
| **内部审计** | 组织内部审计人员 | 面向内部受众；汇报路线与被评估职能部门完全独立 |
| **外部审计** | 外部审计公司 | 理论上无利益冲突，具有高度外部有效性 |
| **第三方审计** | 监管机构或其代表 | 发起方选择审计师并设计范围，**被审计企业无法控制该过程** |

#### SOC审计标准与报告 (SSAE 18 / ISAE 3402)
**通用标准**：系统和组织控制(SOC)审计，允许通过一次外部评估生成报告分享给客户，代替多次繁琐的第三方评估。

**SOC审计分类**：
- **SOC 1**：专门评估**影响财务报告准确性**的控制措施。
- **SOC 2**：评估影响系统内信息的安全性（CIA）及隐私的控制措施。结果**机密**，仅在NDA下共享。
- **SOC 3**：评估内容同SOC 2，但结果**旨在向公众公开披露**。

**SOC报告类型**：
- **Type I**：仅涵盖**特定时间点**，侧重文档化审查，提供对控制措施描述及设计适用性的意见。
- **Type II**：涵盖**至少6个月**的延长运营期，不仅评估设计，还验证控制措施的**运行有效性**。

⚠️ **考试陷阱**：Type I 看的是"设计"和"时间点"，Type II 看的是"运行有效"和"时间段（≥6个月）"。

#### 审计框架
- **COBIT (信息和相关技术控制目标)**：常用于审计和评估的框架，描述组织在其信息系统周围应具备的通用控制要求。

### 15.2 开展漏洞评估

#### 漏洞评估基础
- **漏洞评估**：使用自动化手段检查系统弱点，制定修复路线图以消除不可接受风险。
- **SCAP (安全内容自动化协议)**：NIST提供的通用框架，提供通用语言描述/评估漏洞，促进安全系统交互自动化。

#### SCAP组件 (记忆法：**C-C-C-C-X-O**)
| 缩写 | 全称 | 用途 |
| :--- | :--- | :--- |
| **CVE** | 通用漏洞披露 | 漏洞的**命名**系统 |
| **CVSS** | 通用漏洞评分系统 | 漏洞**严重程度**的评分系统 |
| **CCE** | 通用配置枚举 | 系统配置问题的**命名**系统 |
| **CPE** | 通用平台枚举 | OS/应用/设备的**命名**系统 |
| **XCCDF** | 可扩展配置检查表描述格式 | 指定安全检查表的**语言** |
| **OVAL** | 开放漏洞与评估语言 | 描述安全测试程序的**语言** |

⚠️ **区分点**：CVE给漏洞起名字，CVSS给漏洞打分数；XCCDF写检查表，OVAL写测试程序。

#### 网络扫描技术
- **网络发现扫描**：搜索开放网络端口，**不探测**具体漏洞。
- **TCP SYN 扫描**：又称"半开"扫描，发送SYN包请求连接。
- **TCP Connect 扫描**：打开完整连接，用于**没有权限**执行半开扫描时。
- **TCP ACK 扫描**：发送ACK包，用于**探测防火墙规则**及方法论。
- **UDP 扫描**：使用无连接UDP协议检查活动UDP服务。
- **圣诞树扫描**：同时设置**FIN、PSH和URG**标志位。

💡 **记忆法**：圣诞树扫描就像圣诞树上同时亮起三种颜色的灯（FIN/PSH/URG全置1）。

#### Nmap 端口状态
| 状态 | 含义 |
| :--- | :--- |
| **Open** | 端口开放，应用程序主动接受连接 |
| **Closed** | 端口可访问（防火墙放行），但无应用程序响应 |
| **Filtered** | 防火墙干扰，无法确定开放/关闭 |
| **Unfiltered** | 端口可访问，但无法得出确定结论 |

#### 漏洞扫描与结果分析
- **横幅抓取**：读取服务欢迎屏幕信息以识别服务变体和版本。
- **网络漏洞扫描**：利用已知漏洞数据库探测目标。
- **误报**：没有问题却报告有漏洞。
- **漏报**：有漏洞却未能发现。
- **未认证扫描**：无特权，从攻击者视角运行，评估不全面。
- **认证扫描**：具有只读权限，可读取配置，**提高准确性，减少误报和漏报**。
- **Web漏洞扫描**：操纵输入等参数探测Web应用漏洞。
- **数据库漏洞扫描**：扫描影响数据库安全的漏洞（如SQL注入）。

#### 渗透测试与团队
- **渗透测试**：实际采取攻击行动击败安全控制，超越单纯漏洞扫描。
- **白盒测试**：提供详细信息，跳过侦察（已知环境）。
- **灰盒测试**：提供部分信息，平衡黑白盒优缺点（部分知识）。
- **黑盒测试**：不提供信息，模拟外部攻击者侦察（未知环境）。
- **BAS (入侵和攻击模拟)**：自动化渗透测试，注入威胁指标（如放置可疑文件/发送信标数据包）触发测试。
- **合规性检查**：验证控制措施是否满足监管要求。

### 15.3 测试软件

#### 代码审查与测试方法
- **代码审查/同行审查**：由非编写者审查代码缺陷（软件评估基础）。
- **Fagan检查**：最正式的代码审查，严格六步过程，有明确的进入和退出标准。
- **静态应用安全测试 (SAST)**：**不运行**软件，分析源代码或编译后程序（白盒）。
- **动态应用安全测试 (DAST)**：在**运行时**环境中评估，无需源代码（黑盒）。
- **交互式应用安全测试 (IAST)**：在运行时内部进行实时分析（结合了SAST和DAST特点，监控行为/流量/组件）。
- **运行时应用自我保护 (RASP)**：运行在服务器上，**拦截**进出调用并验证数据请求（具有主动拦截防御能力）。

⚠️ **考试陷阱**：SAST是不运行的静态分析，DAST是运行时的外部黑盒测试，IAST是运行时的内部透视测试，RASP不仅是测试工具更是拦截防护工具。

#### 软件测试与监控技术
- **综合交易**：运行脚本化已知预期结果的交易，比对输出寻找缺陷。
- **基准测试**：用预定义标准（响应时间/吞吐量）衡量性能。
- **模糊测试**：提供随机/无效输入以触发极限和缺陷。
  - **变异模糊测试**：基于已有输入进行变异修改。
  - **生成型模糊测试**：基于数据模型创建全新输入。
- **接口测试**：评估不同模块协同工作能力。
- **误用案例测试**：枚举已知误用/滥用案例并尝试利用。
- **测试覆盖率分析**：已测试用例数 / 总用例数。
- **道德披露**：先私下报告供应商，给合理时间打补丁，未修复再公开。

#### 监控类型
| 类型 | 机制 | 特点 |
| :--- | :--- | :--- |
| **被动监控** | 捕获网络流量分析 | 提供真实世界数据 |
| **真实用户监控 (RUM)** | 重组用户活动跟踪交互 | 被动监控的变体 |
| **综合监控** | 执行人工/模拟事务 | 在真实用户遇问题前**主动**检测 |

### 15.4 培训和演练

#### 网络安全演练团队
| 团队 | 角色 | 职责 |
| :--- | :--- | :--- |
| **红队** | 攻击者 | 试图获取系统和网络访问权限 |
| **蓝队** | 防御者 | 保护系统/网络，监控环境，实施主动防御 |
| **白队** | 裁判/观察员 | 解决规则争议，记录经验教训，确保不破坏生产环境 |

💡 **记忆法**：红队（像红军一样进攻），蓝队（像蓝盾一样防守），白队（像白袍裁判一样中立监督）。

### 紫队演练与安全演练类型

#### 紫队演练
*   **定义**：在演练结束后，将红队和蓝队聚集在一起共享战术信息和经验教训的过程。通过结合攻防双方的知识来提升整体学习效果。
*   **记忆 aid**：紫 = 红 + 蓝。紫队不是第三支队伍，而是攻防（红蓝）融合交流的机制。

#### 夺旗赛
*   **定义**：一种包含明确预设目标的评分制网络安全演练。红队尝试执行任务（如破坏网站、窃取文件），系统根据红队成功达成的目标数量与蓝队成功阻止的数量进行对比评分。

#### 桌面演练
*   **定义**：一种不使用任何真实系统和设备的讨论型演练。参与者聚集在同一房间内，共同推演和探讨针对虚构安全事件场景的响应步骤。
*   **⚠️ 边缘情况/区分点**：桌面演练是纯理论/讨论性质的，**零**系统影响；而CTF和红蓝对抗涉及真实或模拟的系统操作。

### 15.5 实施安全管理流程和收集安全流程数据

#### 安全管理流程
*   **定义**：旨在验证信息安全程序的持续有效性，并对内部攻击威胁起到威慑作用的管理活动集合。
*   **包含要素**：日志审查、账户管理、灾难恢复和业务连续性、备份验证、培训和意识提升、关键绩效和风险指标的跟踪。

#### 安全信息和事件管理
*   **定义**：SIEM平台通过使用许多设备、操作系统和应用程序中存在的 syslog 功能收集信息，自动化了大部分日常的日志审查工作。

#### 网络时间协议
*   **定义**：日志系统利用NTP确保将日志条目发送到SIEM的系统以及SIEM本身的时钟保持同步，从而保证来自多个来源的信息具有一致的时间线。
*   **⚠️ 常见误区**：NTP的核心目的不仅是"对时"，在SIEM场景下，其**真正价值是保证多源日志的时间线一致性**，以准确还原攻击链。时间不同步将导致日志关联失效。

#### 网络流量日志
*   **定义**：提供了系统之间连接以及传输数据量的记录，在调查安全事件时特别有用。
*   **对比记忆**：系统日志记录“发生了什么操作”，NetFlow记录“谁和谁连接了、传了多少数据”。

#### 账户管理审查
*   **定义**：旨在确保用户仅保留经授权的权限，并防止发生未经授权的系统修改。
*   **核心目标**：防范权限蔓延和特权残留。

#### 备份验证
*   **定义**：管理者定期通过检查日志、验证哈希值或要求对系统或文件进行实际恢复，来验证备份流程是否有效运作并满足组织的数据保护需求。
*   **⚠️ 考试陷阱**：仅有备份是不够的，**未经验证的备份等于无效备份**。验证方式包括：查日志、验哈希、实际恢复测试。

#### 灾难恢复和业务连续性测试
*   **定义**：定期测试这些控制措施，为组织提供有效防范业务运营中断的保障。

#### 钓鱼模拟
*   **定义**：组织使用伪造的钓鱼信息来确定用户是否容易受到攻击，从而评估安全意识程序的有效性。
*   **后续动作**：将点击/响应模拟钓鱼的用户重定向至培训资源，以提升其识别可疑活动的能力。
*   **记忆 aid**：钓鱼模拟的目的不是“惩罚”中招者，而是“诊断+治疗”（评估有效性 + 定向培训）。
